## Executive Summary

The M4 prerequisites branch made SQL repo methods async (correct design), but **did not update consumers**. Test stubs remain outdated. This creates:

1. **Formatting violations** → ruff format wants to reformat 5 files  
2. **Type mismatches** → mypy --strict cannot resolve:
   - Awaiting non-async results (Sheets repo methods return sync values)
   - SQLAlchemy column types lack runtime `.date()` / `.isoformat()` methods
   - Type annotations mismatch between SQLAlchemy columns and Python types
3. **Test failures** → Tests call async methods without `await` when DATABASE_URL is set  

**Root cause:** The branch made **read methods async in SQL repos**, but the **_DualWrite wrappers and all callers still expect sync**.

## Issue 2: Typecheck Failures (19 errors across 6 files)

### Category A: SQLAlchemy Column Type Issues (4 errors)

**Pattern:** Calling `.date()` or `.isoformat()` on SQLAlchemy column types without checking nullability.

| File | Line | Error | Reason |
|------|------|-------|--------|
| `app/repos/sql/maintenance.py` | 59 | `"DateTime" has no attribute "date"` | `performed_at: Mapped[sa.DateTime]` is SQLAlchemy type, not Python datetime. Cannot call `.date()` on it. |
| `app/repos/sql/brew_log.py` | 114 | `"DateTime" has no attribute "date"` | Same as above for `brewed_at: Mapped[sa.DateTime]` |
| `app/repos/sql/inventory.py` | 40 | Assignment type mismatch | Setting `existing.roast_date = _to_date(...)` where `roast_date: Mapped[sa.Date \| None]` — mypy sees `sa.Date` not Python `date` |
| `app/repos/sql/inventory.py` | 96 | `"Date" has no attribute "isoformat"` | Same as above — `roast_date: Mapped[sa.Date]` is SQLAlchemy type |

**Fix strategy:** After `.date()` or before `.isoformat()`, mypy needs an explicit cast to Python `date`. The row object returned from the DB is already deserialized to Python types at that point, so the cast is safe but needed for mypy --strict.

### Category B: Incorrect Type Annotation in Catalog Repo (1 error)

**Error:** `app/repos/sql/catalog.py:77`
```
Function "app.repos.sql.catalog.SqlCatalogRepo.list" is not valid as a type
```

**Root cause:** Line 77 references `_fetch_all` method which aliases to `list()`. The method is declared as:
```python
async def _fetch_all(self) -> list[dict[str, Any]]:
    return await self.list()
```

The issue is that the **return type annotation is bare `list`** instead of using the full qualified form. Should be `builtins.list[dict[str, Any]]` to be consistent with other repos and avoid collision with the method name `list()`.

**Fix:** Change return type annotation in `_fetch_all()` to `builtins.list[dict[str, Any]]`.

### Category C: Incompatible "await" — Sheets vs SQL Repo Mismatch (14 errors)

**Pattern:** Routers and services are awaiting methods that may return sync values.

**Example:** `app/routers/api_catalog.py:119`
```python
return [_catalog_to_out(row) for row in await catalog_repo.list()]
```

But `catalog_repo` is `_DualWriteCatalogRepo`, and its `.list()` sometimes returns sync (Sheets) and sometimes async (SQL):

```python
async def list(self) -> builtins.list[dict[str, Any]]:
    if settings.use_postgres and self._sql is not None:
        return await self._sql.list()   # SQL is async ✓
    return self._sheets.list()          # Sheets is sync — NO AWAIT ✗
```

When `use_postgres=False` (default, local testing), the method returns a sync value without `await`. But the caller always uses `await`, creating a type violation.

**Files affected (14 errors across these locations):**
- `app/services/inference.py`: 3 errors (lines 248, 258, 280)
- `app/services/defaults.py`: 8 errors (lines 71, 89, 96, 100, 105, 107, 115, 119)
- `app/routers/api_maintenance.py`: 4 errors (lines 44, 52, 75, 94)
- `app/routers/api_inventory.py`: 6 errors (lines 21, 48, 50, 52, 67, 93)
- `app/routers/api_hardware.py`: 4 errors (lines 75, 85, 90, 171)
- `app/routers/api_dashboard.py`: 3 errors (lines 27, 30, 33)
- `app/routers/api_catalog.py`: 11 errors (lines 119, 131, 136, 141, 152, 221, 306, 421, 431, 456, 464)
- `app/routers/api_brew_log.py`: 7 errors (lines 56, 59, 62, 127, 138, 153, 203)

**Root cause:** The _DualWrite wrappers were updated to be async but the **Sheets repo methods remain sync**. This creates an inconsistency: the wrapper sometimes awaits, sometimes doesn't.

**Fix option 1 (recommended by design):** Make all Sheets repo methods async to match SQL repos. The wrappers can then consistently `await` both. This requires updating all read method signatures in:
- `app/repos/sheets/catalog.py` → CatalogRepo
- `app/repos/sheets/inventory.py` → InventoryRepo
- `app/repos/sheets/hardware.py` → HardwareRepo
- `app/repos/sheets/maintenance.py` → MaintenanceRepo
- `app/repos/sheets/brew_log.py` → BrewLogRepo

And update the wrappers to always `await`.

**Fix option 2 (not recommended):** Make SQL repos return sync values. This defeats the purpose of async/await and will cause issues under concurrent load.

## Issue 3: Test Failures (16 tests failing, all in existing tests — NOT the new test_sql_repos_read.py)

### Why Tests Pass Locally But Fail in CI

**Local behavior:**
- No `DATABASE_URL` env var is set by default
- Test fixture `conftest.py` (in `tests/repos/sql/`) skips the entire test module:
  ```python
  if not os.getenv("DATABASE_URL"):
      pytest.skip("DATABASE_URL not set — skipping SQL repo integration tests")
  ```
- Tests are silently skipped → no failures

**CI behavior:**
- GitHub Actions workflow sets `DATABASE_URL` for the PostgreSQL service
- Tests now run (are NOT skipped)
- Tests call async methods without `await`:
  ```python
  async def test_list_returns_empty(db_session: AsyncSession) -> None:
      repo = SqlInventoryRepo(db=db_session)
      assert repo.list() == []  # BUG: repo.list() is async, needs await
  ```
- Comparing a coroutine object to `[]` → AssertionError

### Affected Test Files (6 files, 16 total failures)

| File | Failing Tests | Count | Issue |
|------|---------------|-------|-------|
| `tests/repos/sql/test_inventory.py` | All 6 tests | 6 | Call `.list()`, `.list_all()`, `.get()` without `await` |
| `tests/repos/sql/test_brew_log.py` | 2 tests | 2 | Call async methods without `await` |
| `tests/repos/sql/test_catalog.py` | 3 tests | 3 | Call `.list()` and `.get()` without `await` |
| `tests/repos/sql/test_hardware.py` | 2 tests | 2 | Call async methods without `await` |
| `tests/repos/sql/test_maintenance.py` | 2 tests | 2 | Call async methods without `await` |
| `tests/repos/sql/test_dual_write.py` | 1 test | 1 | Assertion on unwrapped coroutine |

**Note:** `tests/repos/test_sql_repos_read.py` (new test file added in this PR) has **all tests passing**. This is the litmus test — if the stubs are fixed, the read tests will continue to pass.

**Fix:** Update all test calls to use `await` for async methods:
```python
# Before:
assert repo.list() == []

# After:
assert await repo.list() == []
```

**Finding:** The code was pushed without running `mypy --strict app/`. If this had been run before push, all 19 errors would have been caught.

**Inviolable Rule 3 (CI Discipline):** Build failures trigger root cause analysis. This failure should have been caught pre-push by the developer running checks locally.

**Recommended process fix:**

- Document in CONTRIBUTING.md that all PRs must pass `uv run mypy app/ --strict` before push

## Executive summary
Run 26736087268 failed for five independent reasons across test, dependency audit, security scanning, Semgrep, and Gitleaks. The failures are not caused by lint, format, mypy, frontend lint/test/build, or coverage threshold.

## Failed jobs and root causes

### 1) `test` job — async event-loop mismatch in SQL repo tests
**Observed failure**
- Job: `test`
- Failing tests begin in `tests/repos/sql/test_household_repo.py`
- Primary error from run log:
  - `RuntimeError: Task ... got Future ... attached to a different loop`
  - surfaced while `test_create_household_creates_admin_member` calls `UserRepo.create(...)` and `await db.flush()`
  - traceback reaches asyncpg transaction startup during nested transaction/savepoint creation

**Evidence**
- `tests/repos/sql/test_household_repo.py:35-55` marks the repo tests with `@pytest.mark.anyio`
- `tests/repos/sql/conftest.py:50-82` provides async engine/session fixtures and uses nested transaction/savepoint isolation
- `pyproject.toml:44-45` enables `pytest-asyncio` auto mode

**Root cause assessment**
The SQL repo test harness is mixing async test/fixture lifecycle mechanisms in a way that allows asyncpg objects to be used from a different event loop than the one that created them. The CI traceback points to savepoint/transaction startup, which is consistent with loop affinity problems between the anyio-marked tests and the async SQLAlchemy/asyncpg fixtures.

**Impact**
This is a real test harness/runtime issue, not a coverage or assertion problem. The failing job otherwise reached `81.67%` coverage in CI, so the `test` job is blocked by the loop mismatch itself.

### 2) `dependencies` job — known vulnerable Starlette version
**Observed failure**
- Job: `dependencies`
- Command: `uv run pip-audit --ignore-vuln PYSEC-2025-185`
- Audit result:
  - `Found 1 known vulnerability in 1 package`
  - `starlette 1.0.0   PYSEC-2026-161 1.0.1`

**Root cause assessment**
The lockfile/environment resolves `starlette==1.0.0`, and the CI dependency audit treats `PYSEC-2026-161` as actionable because only `PYSEC-2025-185` is ignored in workflow config.

**Impact**
The job is expected to fail until the dependency graph no longer resolves to the vulnerable Starlette version or the audit policy changes.

### 3) `security` job — Bandit flags dynamic SQL in household backfill
**Observed failure**
- Job: `security`
- Command: `uv run bandit -r app/ -ll`
- Finding:
  - `B608:hardcoded_sql_expressions`
  - Location: `app/repos/sql/household.py:496`

**Evidence**
`app/repos/sql/household.py:493-499` builds SQL with:
```python
sa.text(
    f"UPDATE {table} SET household_id = :hid WHERE household_id IS NULL"
)
```

**Root cause assessment**
Bandit treats the f-string passed into `sqlalchemy.text(...)` as a potential SQL injection vector. The inline `# noqa: S608` suppresses Ruff but does not suppress Bandit, so CI still fails on this construct.

### 4) `semgrep` job — same dynamic SQL pattern blocked by Semgrep
**Observed failure**
- Job: `semgrep`
- Finding:
  - `python.sqlalchemy.security.audit.avoid-sqlalchemy-text.avoid-sqlalchemy-text`
  - File: `app/repos/sql/household.py`
  - Blocking lines: `495-497`

**Root cause assessment**
Semgrep independently rejects the same `sqlalchemy.text(f"UPDATE {table} ...")` pattern. This is a second security gate failing on the same implementation choice as Bandit, so even if one scanner were waived, the other would still block CI.

### 5) `gitleaks` job — JWT secret-like literals committed in tracked files
**Observed failure**
- Job: `gitleaks`
- Tool reported `leaks found: 3`

**Findings called out in the run log**
- `tests/test_integration.py:17`
- `tests/conftest.py:14`
- `.squad/agents/alex/history.md:148`

**Evidence**
- `tests/conftest.py:13-14` sets `JWT_SECRET` to a fixed 32-character literal
- `tests/test_integration.py:15-18` documents a run command with the same literal secret
- Gitleaks classifies these as `generic-api-key`

**Root cause assessment**
The repository contains committed secret-shaped literals for `JWT_SECRET`, including one in a `.squad` history artifact that is part of the PR diff. Gitleaks scans commit history/diff content, so these literals are treated as secret leaks even if they are test-only values.
